# Helper 项目 — 重启版

> 旧版本(鳕鱼助理 / PM 对话助手)已全部清空。本项目从零开始,方向重定。

---

## 一、项目方向

**做内部信息化团队的"数字员工"。先做我们自己岗位 / 我自己的平替,作为可复制方法论的 dogfooding 样本。**

### 为什么不再做之前那个

- 之前是"对所有 PM 都有用的对话助手",通用能力被开源 agent(Claude Code / OpenCloud / Cursor)的迭代速度抹平。
- 通用 agent 框架不是内部信息化团队该有的护城河。

### 为什么是"自己岗位"

- 我们团队每天处理的工单 / 异常 / 跨系统请求,本质就是 agent 最适合的工作:开放式、跨系统、需要判断、量大。
- 数据天然在我们手里(工单系统就是训练材料)。
- 用户就是自己 + 全公司发工单的人,无跨部门协调成本。
- 跑通后,横向复用给法务工单 / HR 工单 / 财务工单 同构场景。

---

## 二、核心架构纪律

**agent 运行时是消耗品,不在这层和开源比。沉淀放在三件套。**

```
[运行时层]    Claude Code / OSS agent — 可替换,永远用最强的
     ↓
[标准接口]    MCP / OpenAPI — 不发明新协议
     ↓
[护城河三件套]
  ① 内部系统工具集 — 工作中碰到的每个系统的 MCP / API 化
  ② 个人 context 库 — 决策记录、判断标准、偏好规则、踩过的坑
  ③ 工作流剧本 — 高频任务的参数化 playbook
[触发层]      工单 / IM @ / 告警 / 邮件 → agent 主动激活,不等用户来聊
```

OSS agent 出新版,换上面那层,下面三件套不动。

---

## 三、两个待深入的核心问题

这两件事工业界今天都没解,是 v1 必须想清楚的:

### 1. 方法论怎么从"我"扩展到其他岗位

- 直接抄 playbook 不行,隐性知识不可移植。
- 思路:做"构建方法论的方法论"——schema 层(playbook / 决策日志 / 偏好规则的数据结构) + bootstrapping agent(shadow 新岗位 2 周,主动追问,把脑里的东西外化进 schema)。
- 我们的数字员工本身是 reference implementation。

### 2. 长期记忆怎么可靠维护

- Claude Code / Codex 的 memory 不可靠,是结构问题——所有东西塞一个桶按相似度召回必然崩。
- 思路:
  - 分层存储(事实 / 决策日志 / 偏好规则 / 模式知识 / 工作记忆),每层独立写入和失效策略。
  - event sourcing:权威源是 append-only 决策日志,模式知识从日志重新蒸馏(不增量维护)。
  - 人在 loop:每周 15 分钟 review agent 蒸馏出的新规则,确认 / 否决 / 编辑。
