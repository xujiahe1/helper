# 产品逻辑(给产品看的版本)

> 这份文档不写一行技术语言。技术细节在 architecture.md / runtime.md / roadmap.md,你不用看。

---

## 一句话:我们在做什么

把企业里某个专家**脑子里靠经验做的判断**,变成一份**其他 agent 能直接照着干的"操作手册"**。

最终结果: 你对着这个数字大脑提问,它的回答**和那个专家本人差不多**——而且会告诉你它依据了什么、有多确定、什么时候应该回头叫真人。

**不是再造一个 ChatGPT**——通用大模型一抓一大把。
**是造那份"操作手册"——这才是真东西**。它持续生长、有版本、能跨人读、能直接喂给 agent 执行。

---

## 这个产品本质是个"业务流水线"

专家的脑子 → 经过 5 道工序 → 数字大脑

```
专家随手一句话      整理成结构化判断       识别出共同模式      
   "L0"      ──→       "L1"          ──→     "L2"        
                                                      ↓
   能让 agent 直接干 ←── 把规约 + 反例打包 ←── 主动追问把边界问出来
        "L4"               "L3 ★ 灵魂"        
                              ↑ 这一步是产品命门
   ↓
   agent 跑历史 case → 不一致回流给专家 → 持续修正  "L5"
```

### 五道工序具体讲

1. **接原料**: 专家随便扔——一句话、一段语音、几篇 wiki 文档,都吃。我们后台把它**结构化**: "什么场景下、看到什么信号、权衡了什么、最后选了什么、为什么没选别的"
2. **找模式**: 同类的判断攒到几条,识别出"哎这其实是同一种规则",自动起草一条**候选规约**,专家点头就入库
3. **画边界 ★**: 产品**主动追问**专家——"那如果场景反过来呢?"、"信号 X 变了你还这么判吗?"、"过去有没有反例?"——把专家自己都没意识到的边界条件**问出来**
4. **打包**: 规约 + 反例 + 关联事实 → 编译成 agent 能直接读的格式
5. **回炉**: agent 拿这套规约去 replay 历史 case,跟专家实际判断比对,不一致的地方回流给专家 review

### 为什么第 3 步是命门

业界做"专家数字人"的失败案例(Character.ai 风格 / 商汤如影 / 各家企业 KMS+LLM)**全卡死在这一步**: 他们都试图从"已经存在的文档/聊天记录/录屏"里挖知识——挖不出来,因为**专家的判断不在纸上,在脑子里**。

我们的差异化是: **在判断发生的当下,通过对话把它问出来**。

而这个"怎么问、什么时候问、问什么"是有积累价值的产品 IP——不是写一次 prompt 就完事的事,需要持续磨。**别家想抄,得花同样多时间从头攒经验**。

## 用户能摸到的 5 件事

不管用户走 IM 还是后台,体验是 5 个面:

| # | 名字 | 比方 | 干啥 |
|---|---|---|---|
| 1 | **扔东西窗口** | 像微信对话框 | 专家随手扔判断/语音/文档,bot 不挑姿势 |
| 2 | **我的待办** | 像 Gmail 收件箱 | 每周 30 分钟来这清——分四档:冲突要裁决 / bot 要追问的边界 / 候选规则等点头 / 文档抽出的批量确认 |
| 3 | **知识库浏览** | 像翻 wiki | 看当前数字大脑里有啥规约、啥事实、啥 case,每条都点得开看出处 |
| 4 | **问数字大脑** | 像 ChatGPT | 但每个回答都带**依据 + 不确定性**,bot 知道自己什么时候不行,会主动让真人接手 |
| 5 | **冲突仲裁台** | 像 PR review | 两个专家意见对不上时,把吵架过程录下来,最终留一条主版本 + 反例附注 |

---

## 多个专家会不会打架

不会——因为我们设计上就**不允许"张三视角"和"李四视角"并存**。

- 数字大脑里只有**一条主版本**的规约
- 张三和李四谁说的,我们记在底层(可追溯,审计能查),但**呈现给 agent 的只有"目前共识 + 历史反例"**
- 吵架过程本身也变成知识——下次类似分歧,bot 知道上次怎么解决的

---

## 凭什么这事不会被 OSS Agent 框架吃掉

押 5 件事,翻译成业务话术:

1. **数字大脑长在 git 仓库里,而不是某家公司的 SaaS 数据库** → 专家能直接读、能用 git 看 diff、新岗位接入直接 git clone 复制走
2. **不预设领域结构** → 没有"工单领域 schema"或"HR 领域 schema",所有概念都是从专家说话里**自己长出来的**——这是我们坚持"领域无关"的根
3. **追问怎么问是有积累的产品 know-how** → 不是 prompt 写一次就完事
4. **不依赖把模型 fine-tune 成专家** → 专家能力全在那份 git 仓库,模型可替换。今年 Opus,明年换 GPT-6,数字大脑不动
5. **多专家不试图合并成折衷意见** → 只走"主版本 + 反例附注",避免折衷意见的灰度泥潭

## 你最关心的两个问题

### Q1. M1 直接走 IM,不做临时 Web

最初的方案里 M1 用后台 Web 对话框做"用户验收链路",**这个被否了**。理由:没真实用户的时候,做一个用完就扔的 Web 是浪费——直接做最终生产形态(Wave IM)更值。

实算接 IM 总开销 1 周不到(Wave 回调签名/AES 2-3 天 + 部署 1-2 天 + 联调 1-2 天)。M1 4 周里花 1 周做就够,剩 3 周磨流水线本身。

所以 **M1 直接 IM 优先**,Web 浏览界面延到 M3。

### Q2. 怎么知道做对了

| 月 | 验收 | 跑不通就 kill |
|---|---|---|
| **M1**(本地核心闭环) | **你自己** 4 周内扔 30 条判断进去,问 10 个问题数字大脑答对 6 个以上;你能说"下次做这事我会先看一眼规约" | 答对 < 4 / 10 |
| **M2**(加灵魂 + 接 IM) | 追问命中率 ≥ 50%(你觉得问得有道理);规约扩到 80 条;IM @ 能正常对话 | 追问 < 30% 命中 |
| **M3**(复用验证) | 找一个**完全不同领域的专家**(法务/HR/财务),3 周内规约 ≥ 30 条,产品代码改 ≤ 30%(70% 复用) | 第二领域要超 3 周或大量改代码 → **不是产品是定制** |

**M3 是真正决定生死的考试**——这个验收过了,我们就是产品;没过,我们就是个高级定制工具,无法横向复用,直接 kill。

---

### Q3. 知识会不会膨胀几千 MD

**会,如果设计错。我们设计上有硬纪律避免**。

业界大部分企业知识图谱/KMS 的失败模式: 把"所有名词"都建成 entity。工单类型 2000+ / 员工数万 / 系统 500+,几个月就十万级,AI 喂进去不如关键词搜索。

我们的硬纪律 — **只存决策性知识,不存事实性数据**:

| 该存 | 不该存 |
|---|---|
| 决策规约(几百~千) | 工单类型大全 |
| 反例 case(几千) | 员工列表 |
| 决策性事实(几百) | 系统名大全 |
| 核心决策概念(100-300) | 项目列表 |
| | 任何在外部系统就有权威表的内容 |

外部已有权威表的实体,我们只存 ID + 简短描述 + 外部系统的 API 端点,**详细信息即时查,绝不复制**。

加上 4 道防膨胀机制:
1. **合并优先**: 每条新输入抽实体时先找现有近义,有则合并
2. **晋升门槛**: 必须被 ≥ 3 条原料引用 + 关联到规约,才有资格建独立 MD;否则只在原料文本里 inline
3. **周期体检**: 每周 LLM 扫全图,合并近似/归档孤儿
4. **Decay**: 6 月无引用降权(不删,但不进主路径检索)

预估规模: **单岗位 300-1100 个 MD,5 岗位 1300-5000 个**。git + ripgrep + LLM 在 5000 量级运转都很轻松。

膨胀的根因永远是**无脑提取 + 无脑创建**——我们的核心纪律: AI 默认合并、默认复用,**只有专家显式认可才扩张**。
